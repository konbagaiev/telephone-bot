# Spec: Data model and storage

> Roadmap step 1 (`docs/plan.md`). First code in the repository.

## Goal

Define the entities the system operates on — questions, people, assignments,
calls, answers, policy — and where each lives: configuration loaded from YAML,
operational data in Postgres (ADR-016). No network, no telephony, no model.
Everything later in the roadmap reads and writes through this layer.

## Touches

- `pyproject.toml` — project setup and the dependency set (ADR-006, ADR-016).
- `src/config.py` — YAML-backed configuration: `Question`, `Questionnaire`,
  `Policy`.
- `src/models.py` — operational entities: `Person`, `Assignment`, `Call`,
  `Answer`.
- `src/db.py` — connection, session handling, queries.
- `migrations/` — Alembic setup and the first migration.
- `data/example/` — a fictional questionnaire and policy.
- `tests/` — unit tests for the below.

## Does NOT touch

- Telephony, the Realtime session, the audio bridge, the CLI, deployment.
- The GitHub Actions deploy. Migrations must *run*, but wiring them into the
  deploy belongs to roadmap step 2.

## Design notes

Derived from the ADRs; recorded here so the code doesn't have to re-derive them.

**The storage split follows the kind of data, not convenience** (ADR-016).
Questions, questionnaires and policy are configuration: YAML, in git, changed by
diff. People, assignments, calls and answers are operational: Postgres, queried,
never in git. This boundary is the same one `.gitignore` already draws — the
repository is public, and respondent phone numbers are personal data.

**A question is intent, not a script** (ADR-007): an id, the intent, the expected
answer type, whether it is required, and optional per-language phrasing used only
where exact wording matters. The model does the phrasing.

**Configuration references are validated on load.** An `Assignment` in the
database points at a questionnaire id defined in YAML; nothing enforces that
across the boundary, so loading must fail loudly on a dangling reference rather
than at 3 a.m. mid-call.

**An answer is stored twice** (ADR-007): the raw utterance in the language of the
call, plus a normalised value. Without the normalised form, answers in different
languages cannot be compared.

**Three independent status fields** (ADR-005), not one:
- `Call.disposition` — `answered` · `no_answer` · `busy` · `voicemail` ·
  `carrier_failed`
- `Call.end_reason` — `agent_completed` · `agent_stopped` · `remote_ended` ·
  `agent_error`
- `Assignment.status` — `pending` · `in_progress` · `completed` · `partial` ·
  `unreachable` · `opted_out`

**Completion is computed, never asserted** (ADR-002). An assignment is
`completed` when every required question in its questionnaire has an answer. The
model's `end_call` does not decide this; a function over stored answers does.

**People are identified by phone number**, normalised to E.164 on write, with a
uniqueness constraint in the database. Without it the same person under two
spellings becomes two people — and unlike a file, a database can enforce this.

**Policy is parameters only** (ADR-007) — attempts, retry delay, calling window,
maximum call duration, silence timeout, voicemail behaviour, opt-out behaviour.
No recording-disclosure parameter: ADR-014 removed audio retention.

## Acceptance criteria

- `data/example/` holds a valid, fictional example set: a questionnaire of two
  questions and one policy. It loads into validated objects.
- A malformed config file fails with an error naming the file and field, not a
  stack trace from deep inside Pydantic.
- An assignment referencing an unknown questionnaire id is rejected at load.
- The same phone number written in national and E.164 form resolves to one
  `Person` row; a second insert does not create a duplicate.
- Recording an answer persists it and leaves prior answers unchanged.
- Given answers to all required questions, the assignment computes as
  `completed`; missing one required answer computes as `partial`.
- `alembic upgrade head` creates the schema from empty, and `downgrade` reverses
  it.
- `pytest` passes against a local Postgres, with no network access.

## What could go wrong (risks & guards)

- **Risk:** Completion logic silently treats an optional question as required
  (or the reverse), so a questionnaire is marked done while an answer is missing
  — the failure this whole status model exists to make visible.
  **Guard:** Unit tests over the completion function covering all-required-met,
  one-required-missing, and optional-missing.

- **Risk:** Phone numbers in different formats create duplicate people, so
  answers scatter across identities and retries call someone already done.
  **Guard:** Test inserting the same number in two formats and asserting a single
  row; plus a unique constraint so the database refuses it regardless of caller.

- **Risk:** A questionnaire id is renamed in YAML while assignments in the
  database still reference the old one. Nothing in Postgres can enforce a foreign
  key into a file, so this breaks at call time, on a live call.
  **Guard:** Validate all references at load and fail before any call is placed.

- **Risk:** The first migration is written but never verified from an empty
  database, so it works locally against a hand-made schema and fails on the VPS.
  **Guard:** Acceptance criterion above — `upgrade` from empty, then `downgrade`.

- **Risk (accepted, not guarded):** the entity shape turns out wrong once a real
  call runs. That is expected — step 3 exists partly to find out. Note that the
  cost of this is now higher than it was under file storage: fixing it means a
  migration, not editing a file.

## Non-goals

- No CLI. Loading and persistence are library functions; the CLI arrives with the
  runner.
- No answer-normalisation logic. The field exists; filling it is the model's job
  at call time (step 3+).
- No scheduling or retry logic. Policy is *represented* here, not *enforced*.
- No seeding of real people. Only fictional examples.
