# Spec: Admin UI — manage people/assignments and launch calls

> Roadmap step 9. Buildable now: the full call path runs end to end (call 7), so
> ADR-006's "UI last, on top of a system that already runs" is satisfied. Steps 7
> (policy) and 8 (multilingual) stay open — the UI is built on what exists and is
> not coupled to them.

## Goal
A minimal, token-gated web UI inside the existing ASGI app that replaces the two
hand-run operations we do today: seeding the prod DB with a Python snippet, and
dialing with `python -m src.runner`. It lets us add/list/delete people and
assignments, **reset** an assignment to re-run a demo, **launch the next pending
call**, and **view** recorded answers and the stored transcript. Server-rendered
(Jinja, no JS build step).

## Shape decisions (baked in)
- **Launch = next-pending**, reusing the runner: one "Call next" button places the
  oldest pending assignment, exactly as `python -m src.runner` does today. No
  per-person on-demand dial (that widens the surface without demo value; a reset
  makes a specific person pending, and next-pending then picks them).
- **DB ops = create + view + delete/reset.** No editing of existing rows.
- **Access = one shared token in the URL**, checked as a dependency on the UI
  router only. The webhooks (`/voice`, `/stream`) and `/health` stay ungated —
  Twilio requests self-authenticate via signature validation, and adding the token
  there would break every real call.
- **Server-rendered Jinja, no JS build / no HTMX / no CSS framework.** Full-reload
  forms. The repo is Python-only (ADR-001); a frontend toolchain is not worth it
  for a demo admin surface.
- **Reset in place**, not delete-and-recreate: wipe the assignment's answers and
  calls (transcript cascades), set status back to `pending`; the assignment row
  and its id survive (mirrors the `reset_and_call` skill).

These forks (same-app UI · token-in-URL gate with webhooks self-authenticated ·
server-rendered no-build · next-pending launch) are recorded as **ADR-023**, routed
through `docs/plan.md` first per AGENTS.md.

## Touches
- `src/runner.py` — extract `place_next_call(conn, config, carrier, public_base_url)
  -> Call | None` = `validate_references` + `next_pending_assignment` +
  `place_call_for_assignment` (returns None when nothing is pending). `main()` calls
  it; the web handler calls it too. No change to CLI behaviour.
- `src/db.py` — new read/mutate queries, all SQLAlchemy Core like the rest:
  - `list_persons(conn) -> list[Person]`
  - `assignments_for_person(conn, person_id) -> list[Assignment]` (or one joined
    listing for the index — shape decided in implementation)
  - `calls_for(conn, assignment_id) -> list[Call]` (to link each transcript)
  - `delete_person(conn, person_id)` / `delete_assignment(conn, assignment_id)`
  - `reset_assignment(conn, assignment_id)` — delete answers for the assignment,
    delete its calls (transcript cascades), set status `PENDING`. Explicit about
    answers because `answers.call_id` is `SET NULL`, not `CASCADE`.
- `src/app.py` — mount UI routes under a token-gated prefix (e.g. `/ui`), gate as a
  router dependency. Routes:
  - `GET  /ui` — index: people → their assignments + statuses; add-person form;
    per-assignment reset/delete buttons and links; a global "Call next" button.
  - `POST /ui/people` — add a person (+ optional first assignment).
  - `POST /ui/assignments` — add an assignment to an existing person.
  - `POST /ui/people/{id}/delete` · `POST /ui/assignments/{id}/delete`.
  - `POST /ui/assignments/{id}/reset`.
  - `POST /ui/call-next` — place the next pending call via `place_next_call`.
  - `GET  /ui/assignments/{id}` — answers (`raw`/`value`) + computed completion;
    list of calls linking to each transcript.
  - `GET  /ui/calls/{id}/transcript` — the stored segments in order.
  The carrier for `call-next` is obtained via an overridable dependency so tests
  inject a fake `Carrier` (no network), same as the runner tests.
- `src/settings.py` — add `ui_token` (required, no default). **Depends on the active
  settings-centralisation spec (`2026-07-21-1448`).** Recommend landing that first;
  if this lands first, read `UI_TOKEN` via `os.environ` and reconcile when settings
  lands.
- `src/templates/` — new. `index.html`, `assignment.html`, `transcript.html`.
  Minimal, self-contained, no external assets. Via `fastapi.templating.Jinja2Templates`.
- `pyproject.toml` — add `jinja2`.
- `.env.example` (+ `.env.prod` / `update_env`) — document `UI_TOKEN`.
- `docs/architecture.md` — module-map rows for the UI router + templates; a UI note
  in Shape/Call path; the token-gate convention (UI-only, webhooks self-authenticate).
- `docs/plan.md` → `docs/decisions.md` — ADR-023 (see Shape decisions); remove the
  step-9 item / mark the surfaces built.

## Does NOT touch
- **`src/telephony/`** behaviour — the UI dials only through the existing `Carrier`
  and `place_call_for_assignment`; no Twilio type leaks into the UI.
- **Bridge / agent tools / session** — call *content* is unchanged; the UI only
  *initiates* a call and *reads* its results.
- **Launch semantics beyond next-pending** — no on-demand per-person dial, no queue,
  no scheduling, no retries/calling-window (that is policy, step 7).
- **Editing** people/assignments — create, delete, reset only; no update forms.
- **Auth beyond the shared URL token** — no accounts, sessions, roles, or audit log.

## Acceptance criteria
`.venv/bin/python -m pytest` green, `ruff` clean, and:
- **Token gate**: `GET /ui` without the token → 401 (or 404); with the token → 200
  and lists a seeded person's phone. `GET /health` and `POST /voice` are reachable
  without the token (a genuine Twilio call is unaffected — `/voice` still returns
  TwiML-or-403-by-signature, never the UI's 401).
- **Add**: `POST /ui/people` with a national-format phone creates a person with the
  E.164-normalised number and (if requested) a `pending` assignment; the index then
  shows them.
- **Reset**: given an assignment with a finished call, answers, and transcript,
  `reset_assignment` leaves the assignment row with status `pending` and **zero**
  answers, calls, and transcript segments; a *second* assignment's data is untouched.
- **Delete**: deleting a person removes their assignments, calls, answers, and
  transcript (cascade); deleting an assignment removes its calls and answers.
- **Call-next**: `POST /ui/call-next` with an injected fake `Carrier` places a call
  for the oldest pending assignment, writes the Call row with the carrier id, and
  moves the assignment `pending → in_progress` — asserted against the fake, no
  network (mirrors the existing runner test). A **second** immediate `call-next`
  against the same single pending assignment places **no** further call (the
  `pending → in_progress` transition already guards double-dial).
- **View**: `GET /ui/assignments/{id}` shows each answer's `raw`/`value` and the
  computed completion status; `GET /ui/calls/{id}/transcript` shows the segments in
  stored order.
- **Live smoke (manual, not CI)**: on the deployed UI, add yourself, click "Call
  next", answer the phone, then see the recorded answer and the transcript appear.

## What could go wrong (risks & guards)
- **Risk: the token gate lands on the webhooks (Twilio `/voice` starts 401ing and
  every call breaks) or fails to land on the UI (the prod DB and paid dialing are
  world-open on a public URL).** → Guard: gate is a dependency on the UI router
  only; a test asserts `/ui` 401s without the token *and* `/voice` is reachable
  (returns 403-by-signature, not the UI 401).
- **Risk: `reset_assignment` deletes across the wrong boundary** — wipes another
  assignment's data, or leaves orphaned transcript rows because it deleted answers
  but forgot the `SET NULL` calls path. → Guard: a DB test seeds two assignments
  each with a call, answers, and transcript, resets one, and asserts exactly that
  one is emptied and the other is intact.
- **Risk: launching from a web request double-dials** (double-click → two calls, or
  a slow synchronous Twilio REST call blocks the worker). → Guard: `place_next_call`
  reuses `place_call_for_assignment`, which moves `pending → in_progress` in the
  same transaction, so a second click finds nothing pending — covered by the
  double-`call-next` acceptance test. The REST dial only *initiates* (returns fast);
  the call itself proceeds over webhooks.

## Non-goals
- Per-person on-demand dialing, scheduling, retries, calling windows (policy, step 7).
- Editing entities, multi-user auth, sessions, roles, audit log.
- Any styling framework, JS build, or HTMX; live/real-time updates (reload-based).
- Repointing `raw` at the transcript, or surfacing per-question refusal as a
  distinct state (steps 5 / 11) — the view shows what is stored today.
